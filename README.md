# Voice Beats

いつでもどこでも手軽にビートメイクができる、Webブラウザ完結型のステップシーケンサー・サンプラーです。

以下のURLからすぐに体験していただけます：https://seeyoumayday.github.io/VoiceBeats-main/

---

## 1. 作成物の説明

### 何を作ったのか
`Voice Beats` は、Webブラウザ上で直感的にドラムパターンなどのリズムを作成できるステップシーケンサー（サンプラー）です。
画面上のステップボタンをクリックすることで、キック、スネア、クラップ、ハイハット、ベルの5つのトラックを組み合わせたビートパターンを作成できます。
また、各トラックにはデフォルトで用意された音源に加え、PCやスマートフォンのマイクから自分の声をリアルタイムに録音（サンプリング）してステップに組み込める「ボイスサンプリング機能」を搭載しています。作成したリズムパターンは、WAV形式の音声ファイルとしてローカルにダウンロード可能です。

### 目的・背景
「いつでもどこでも作曲を」というモットーのもと、高価なDAW（音楽制作ソフト）や専用のサンプラーなどの機材、専門的な音楽知識がなくても、インターネット環境とブラウザさえあれば、誰でも簡単かつ直感的にビートメイクを楽しめる環境を提供することを目的に制作しました。
デフォルト音源には、開発者自身の肉声（ボイスパーカッション）を加工した親しみやすくユニークなサウンドを採用しており、ユーザー自身が声を吹き込むことで、世界に一つだけのパーソナルなビートボックスを作成できる楽しさを重視しています。

### 開発の経緯（GitHub Pagesへの移行）
本プロジェクトは、もともと1年前はNode.js（Express）をバックエンドとするWebアプリケーションとして動作していました。しかし、より手軽にアクセス・体験でき、デプロイや共有を簡単に行えるようにするため、リポジトリ自体の構成を見直し、GitHub Pagesなどの静的サイトホスティングサービス上でも動作するクライアントサイドで完結するアプリへと移行しました。

## 2. 自身が担当した役割（実装箇所・担当工程）

個人開発として、企画・設計からUI/UXデザイン、フロントエンド・バックエンドの実装、デプロイ用サーバー設定までの全工程を担当しました。

- **担当工程**: 要件定義、UI/UXデザイン、フロントエンド実装、オーディオエンジン設計、メディア処理実装、簡易配信用サーバー構築
- **具体的な実装箇所**:
  - **フロントエンド（UI/UX）**: HTML5/CSS3によるドラムシーケンサー風UI、ステップボタン、各種調整スライダー（音量・ピッチ）、ドラッグ可能な波形トリミングハンドルの実装。
  - **オーディオロジック**: Web Audio APIを用いた各トラック音声の読み込み・デコード（`AudioContext`）、ピッチ（再生速度）および音量のリアルタイム制御。
  - **波形表示・可視化**: Canvas APIを用いた音声波形（Waveform）の非破壊トリミングUI、再生中のオーディオ波形オシロスコープおよび周波数スペクトラム表示のリアルタイム描画。
  - **サンプリング（録音）機能**: Media Recorder APIを用いたマイク入力の取得と、録音データの瞬時バッファ化（各トラックへのアサイン）。
  - **ミックス・エクスポート機能**: Web Audio APIの出力ストリームをキャプチャし、`ffmpeg.wasm` を用いてブラウザ上（クライアントサイド）でWebMからWAV形式に変換・ダウンロードさせる処理の実装。
  - **サーバーサイド**: Expressを用いた静的ファイル配信と、`SharedArrayBuffer` 利用のためのセキュリティヘッダー（COOP/COEP）のミドルウェア設定。

## 3. 直面した課題と解決方法

### 課題①：GitHub Pagesへの移行に伴う `ffmpeg.wasm` のセキュリティ制限（SharedArrayBuffer）
- **問題**: アプリケーションをGitHub Pagesへ移行するにあたり、最大の技術的ハードルとなったのがブラウザのセキュリティポリシー（Spectre脆弱性対策）です。`ffmpeg.wasm` の動作に必須となる `SharedArrayBuffer` を有効化するには、サーバーからCOOP/COEPヘッダーを返す必要があります。しかし、GitHub Pagesのような静的ホスティング環境ではサーバーヘッダーをカスタマイズできないため、ブラウザ上でデコード・変換機能が動かなくなる問題が発生しました。
- **解決方法**: 
  GitHub Pagesなどの静的ホスティング環境や一般的なローカル静的サーバー環境でも正常動作させるため、サービスワーカー（`coi-serviceworker.js`）を導入しました。これにより、ブラウザ側でリクエストとレスポンスをインターセプトして動的にCOOP/COEPセキュリティヘッダーをエミュレート（付与）する仕組みを構築し、サーバー側のヘッダー制御が不要な環境でも `ffmpeg.wasm` が動作するように解決しました。

### 課題②：ブラウザ録音形式（WebM）から汎用的なオーディオ形式（WAV）への変換
- **問題**: ブラウザ標準の `MediaRecorder` でシーケンサーのマスター音声を録音する場合、保存できるフォーマットが `.webm`（あるいは `.ogg`）等の限られた形式のみであり、一般的なプレイヤーで再生可能な `.wav` 形式で直接エクスポートすることができませんでした。
- **解決方法**: Web Audio APIの `MediaStreamAudioDestinationNode` を用いて、シーケンサー全体の再生音をミックスした音声ストリームを作成し、それを `MediaRecorder` で一旦 WebM 形式としてキャプチャしました。その後、WebAssembly版のFFmpegである `ffmpeg.wasm`（`@ffmpeg/ffmpeg`、`@ffmpeg/core`）をブラウザ上で実行し、サーバーを介さずにクライアントのローカル環境だけで WebM から WAV へのデコードおよび変換（エンコード）を行い、WAV形式でのダウンロードを可能にしました。

### 課題③：非破壊的な音声切り出し（トリミング）UIと再生オフセット制御の同期
- **問題**: 音声ファイルの任意の位置から再生を開始・終了させるオフセット制御を、視覚的に分かりやすく操作できるようにする必要がありました。しかし、Canvas上の物理的なピクセル座標、CSSの配置位置、オーディオバッファの長さ（秒単位）の3者を矛盾なくリアルタイムに同期させる処理が複雑でした。
- **解決方法**: Canvasの全横幅に対する左右のハンドルのピクセル位置から、音声バッファの全体長（`duration`）に対する割合（0.0〜1.0）を動的にマッピングする計算式を実装しました。ドラッグ時にCSSの `left`/`right` プロパティとオーバーレイの幅を更新しつつ、内部の `offset` 値を更新。再生時には `AudioBufferSourceNode.start(0, startOffset, duration)` を用いて指定された範囲のみを切り出して再生させることで、直感的かつ非破壊的にサウンドをカスタマイズできるトリミングUIを完成させました。

## 4. 技術情報

### 使用技術・API
- **フロントエンド（コア）**: Vanilla HTML5, CSS3, JavaScript (ES Modules)
- **Web Audio API**: 音声デコード（`decodeAudioData`）、再生制御（`AudioBufferSourceNode`）、音量制御（`GainNode`）、リアルタイムオーディオ解析（`AnalyserNode`によるオシロスコープ・スペクトラム描画）、マスターミックス録音（`MediaStreamAudioDestinationNode`）
- **Media Capture and Streams API / Media Stream Recording API**: マイクからの音声録音（`getUserMedia` / `MediaRecorder`）
- **Canvas API**: トラックごとの音声波形およびオーディオビジュアライザの描画
- **ffmpeg.wasm (`@ffmpeg/ffmpeg` & `@ffmpeg/core`)**: クライアントサイドでの WebM -> WAV フォーマット変換
- **coi-serviceworker**: Service Workerを用いたCOOP/COEPヘッダーの強制付与（セキュリティ制限のバイパス）

### アーキテクチャ
- 完全にクライアントサイド（ブラウザ）側で動作し、すべての音声信号処理、録音、エンコード、波形描画をリアルタイムに処理するクライアントサイドで完結する型のアーキテクチャを採用しています。これにより、サーバー不要またはオフラインに近い環境でも軽快に動作します。

### 開発支援ツール
- 開発支援ツールとして、Github Copilot、Geminiを使用しました。

## 5. 使い方（ローカル起動方法）

本プロジェクトは完全にクライアントサイド（ブラウザ上）で動作するため、特別なバックエンドサーバーは不要です。ただし、セキュリティ制限（`SharedArrayBuffer`）を回避し、サービスワーカー（`coi-serviceworker.js`）を正しく動作させるためには、ローカルサーバー経由で起動する必要があります（ファイルを直接ブラウザで開くだけでは動作しません）。

### 起動手順（例）

以下のいずれかの方法で簡易ローカルサーバーを起動し、ブラウザでアクセスしてください。

#### 方法1: 起動スクリプトを使用する場合（最も簡単）
プロジェクトルートに配置されている `bin` 内のスクリプトを実行することで、自動的にサーバーの起動とブラウザでの起動が行われます。

- **Mac / Linux の場合**:
  ターミナルを開き、プロジェクトルートから以下のコマンドを実行します。
  ```bash
  ./bin/run
  ```
- **Windows の場合**:
  `bin/run.bat` をダブルクリックして実行するか、コマンドプロンプトで実行します。

#### 方法2: Python を使用する場合
ターミナルで `source` ディレクトリに移動し、以下のコマンドを実行します。
```bash
cd source
python -m http.server 8000
```
起動後、ブラウザで [http://localhost:8000/](http://localhost:8000/) にアクセスします。

#### 方法3: VS Code Live Server などの拡張機能を使用する場合
VS Code を使用している場合は、`source` ディレクトリを開き、拡張機能の「Live Server」などを利用して起動できます。

### 注意事項
ブラウザのセキュリティ制限（Spectre対策）により、`ffmpeg.wasm` を動作させるには `SharedArrayBuffer` の使用許可が必要ですが、本プロジェクトでは `coi-serviceworker.js` が自動的に必要なセキュリティヘッダー（COOP/COEP）をエミュレート（付与）します。そのため、一般的な静的ファイルサーバーであれば特別なサーバー設定なしで動作します。
